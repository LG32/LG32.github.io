面试笔记之http与https的区别

什么是http协议

HTTP协议全称Hyper text Transfer Protocol,翻译过来就是超文本传输协议，位于TCP/IP四层模型中的应用层。

HTTP协议通过请求/响应的方式，在客户端和服务端之间进行通信。

这一切看起来很美好，但是HTTP协议有一个致命的缺点：不够安全。

HTTP协议的信息传输完全以明文方式，不做任何加密，相当于是在网络上"裸奔"。这样会导致什么问题呢？让我们打个比方：

A是服务器，B是客户端，A向B发送请求。

但是，由于传送的信息是明文，这个信息有可能被某个中间人C恶意截获甚至篡改。这种行为叫做 中间人攻击 。

后来为了防止这种事情的发生，A跟B之间在事先就约束了一种 对称加密 的方式，并且约定一个随机生成的密钥。后续的通讯中，信息发送方都使用密钥对信息进行加密，而信息接受方通过同样的密钥对信息进行解密。

但是这样还是不能保证信息的绝对安全。

虽然在后续的通讯中我们使用了加密，但是在第一次约定加密方式的时候使用的还是明文通讯，如果是在第一次的通讯时就已经被拦截了，那么密钥就会泄漏给中间人，中间人仍然可以解密后续的所有通讯内容。

既然对称的不行，那么就使用非对称的加密，为密钥的传输做一层额外的保护。

非对称加密的一组密钥对中，包含了一个公钥和一个私钥。明文既可以使用公钥加密，用私钥解密；也可以用私钥加密，用公钥解密。

在A跟B建立通信的时候，B首先把自己的公钥key1发送给A，A在收到了公钥以后，A自己生成一个用于对称加密的key2，并且用刚才收到的公钥key1对key2进行加密，发给B。B利用自己的非对称加密密钥，解开了公钥key1的加密，获得了key2的内容。从此之后两人就可以使用key2进行对称加密通信了。

但是尽管这样可以防止中间人截取到key1进行解密，但是道高一尺魔高一丈，假如当中间人截取了B给A的密钥key1并替换成key3传给了A，但是由于A不知道公钥被偷偷换过，以为Key3就是B的公钥。于是按照先前的流程，用Key2，发给B。

在这个时候中间人再次截取了，中间人先用自己的密钥解开了key3的加密，获得了key2，然后获得key2，然后再用当初B发来的key1重新加密，再发给B。

这样一来中间人就完全掌握了双方的所有加密的方法了。

HTTPS的解决方法

  1. 作为服务端的B，首先将自己的公钥发给了证书颁发机构，向证书颁发机构申请证书。
  2. 证书颁发机构自己有一对公钥密钥。机构利用自己的密钥来加密Key1，并且通过服务端网站生成一个证书签名，证书签名同样经过机构的密钥加密。证书制作完成后，把证书发送给了服务端的B。
  3. 当A向B再次请求通信的时候，B不再使用的公钥，而是把自己申请的证书发给了A。
  4. 当A收到证书的时候，第一件事就是验证证书的真伪。需要说明的是各大浏览器中已经维护了所有的权威证书的名称和公钥。所以A只要知道这是哪个机构颁发的证书，就可以从本地找到对应的机构公钥，解密出证书的签名。

接下来，A也按照同样的方法自己生成一个证书签名，如果两个签名一致，说明证书是有效的。

  1. 验证成功后，A就可以安心的利用机构的公钥，解密出服务端的公钥key1。并像之前一样，A生成自己的对称加密密钥key2，并用服务端公钥key1加密key2，发给B。
  2. 最后，B用自己的密钥解开加密，得到对称加密密钥key2。于是两人开始使用key2进行对称加密的通信。

在这种方法中为什么没有了中间人攻击呢？

因为证书的签名是由服务端的网址等信息生成的，而且经过了机构密钥进行加密，中间人也无法篡改。所以，发给小灰的假证书是无法验证通过的。

完事！
